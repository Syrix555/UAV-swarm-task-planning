# 毕设后续开发计划

更新时间：2026 年 4 月 28 日  
预计答辩时间：2026 年 5 月底

当前项目已经进入“主链路基本成型，后续以补齐展示闭环、固定实验结果和服务论文撰写为主”的阶段。后续目标不再是继续扩展全部动态场景，而是：

> 稳定交付一条论文可讲、答辩可演示、结果可复现的无人集群协同打击任务规划主链路。

推荐主链路为：

```text
场景加载
-> PSO 任务预分配
-> 输出 UAV 任务序列
-> 动态事件触发
-> MCHA 局部任务重分配
-> 输出重分配后任务序列
-> A* + B 样条/运动学平滑生成最终航迹
-> 输出论文图表和实验指标
```

## 一、当前已经完成的核心工作

### 1. 任务序列建模与预分配改造

状态：已完成主干

当前已经将系统从原来的“一架 UAV 最多执行一个任务点”改造为“每架 UAV 可按 `ammo` 执行一个任务序列”。

已经完成：

- `ammo` 语义调整为可执行任务数量。
- 新增并接入 `AssignmentPlan / UavTaskSequence / TaskNode`。
- PSO 候选解可解码为 UAV 任务序列。
- 代价评估按任务链累计计算：

```text
UAV -> T1 -> T2 -> T3
```

而不是旧的：

```text
UAV -> T1
UAV -> T2
UAV -> T3
```

这部分已经能够支撑论文中“协同打击时间一致性约束”的建模和实验图展示。

## 二、后续最高优先级工作

### 1. 路径规划接入最终任务序列

优先级：最高

路径规划是当前主链路中最需要补齐的部分。虽然路径规划不是论文主要创新点，但题目和论文结构中包含“路径规划与仿真展示”，因此至少需要一张能够说明最终航迹规划结果的图。

建议实现范围：

- 输入重分配后的 `AssignmentPlan`。
- 对每架 UAV 的任务链逐段规划：

```text
UAV 起点 -> T1
T1 -> T2
T2 -> T3
```

- 每段调用现有 A*。
- 对路径做现有 B 样条或运动学平滑。
- 输出最终多 UAV 航迹图。

不建议扩展范围：

- 不要做复杂动态避障。
- 不要将 `THREAT_ADDED` 与 A* 实时绕行深度耦合。
- 不要为了路径规划大改任务分配模型。

### 2. 整理最终主演示脚本

优先级：最高

建议新增或整理一个面向论文和答辩的主演示脚本，用固定场景和固定随机种子跑完整流程：

```text
PSO 预分配
-> 选择代表性动态事件
-> MCHA 重分配
-> 最终航迹规划
-> 输出核心论文图和指标
```

建议主展示事件控制在 3 类：

- `UAV_LOST`
- `TARGET_ADDED`
- `THREAT_ADDED`

`TARGET_DEMAND_INCREASED` 可作为补充实验或备用展示，不建议围绕它继续扩展复杂逻辑。

### 3. 固定最终实验图表和随机种子

优先级：高

当前可视化图已经较多，后续不应继续无限调图，而应尽快固定论文最终使用的图表。

建议固定：

- 一个预分配主场景
- 一个 PSO 消融实验场景
- 三个重分配代表事件
- 一个最终航迹规划展示场景
- 对应随机种子和参数配置

固定后重新生成一版最终结果，后续论文和 PPT 都引用这一版。

### 4. 整理指标表和论文实验文字

优先级：高

建议至少整理以下指标：

- 总规划代价
- 总航程或距离代价
- 威胁代价
- 时间窗惩罚
- 协同到达满足率
- 任务需求满足率
- UAV 平均任务负载
- 重分配变化数量
- 未满足任务需求数量
- 算法运行时间

后续论文中应将这些指标分配到：

- 预分配实验分析
- 重分配事件响应实验分析
- 路径规划展示分析
- 消融实验分析

## 三、可以做，但必须控制范围的工作

### 1. 动图或视频演示

可以做，但应放在静态图、最终航迹和主演示脚本稳定之后。

建议只做轻量演示：

- 将预分配、事件发生、重分配、最终航迹几张关键图串联。
- 或者做简单 GIF，展示任务规划链路变化。

不建议做复杂动画系统，否则容易影响论文写作和最终结果稳定。

### 2. 小幅可视化美化

可以做，但只针对明显影响论文表达的问题：

- 图例遮挡
- 标注过密
- 标题和统计信息留白不合理
- 颜色语义不清晰
- 坐标系内文字遮挡核心信息

不建议为了审美继续大范围重写所有图。

### 3. TARGET_DEMAND_INCREASED 备用展示

可以保留作为补充实验。

只需展示“目标需求增加后，MCHA 补充 UAV 执行任务”，不建议继续围绕该事件扩展复杂机制。

## 四、不建议继续做的工作及原因

### 1. 完整 UavExecutionState 执行态重分配

不建议当前阶段做。

原因：

- 会显著扩大工程量。
- 需要处理当前位置、当前时间、剩余弹药、剩余航程、已完成任务和未完成任务。
- 会影响 `UAV_LOST`、`THREAT_ADDED`、MCHA 代价评估和可视化。
- 容易引入新的边界问题，影响主流程稳定性。

论文中建议表述为：

> 任务序列方案层面的事件触发式局部重分配。

不要表述为已经实现完整执行过程中的实时重规划。

### 2. 任务链中间插入、重排和交换优化

不建议当前阶段做。

原因：

- 当前链尾追加策略已经足够支撑任务序列重分配。
- 中间插入会显著增加代价评估复杂度。
- 链内重排和跨 UAV 交换会影响 PSO、MCHA、测试和可视化。
- 对本科毕设展示收益有限，但实现和调试风险较高。

论文中可以说明采用链尾追加是为了保证重分配快速性和可解释性。

### 3. 多目标同时新增、非连续 id 和删除后复用 id

不建议当前阶段做。

原因：

- 属于工程边界条件，不是论文核心创新点。
- 当前 `TARGET_ADDED` 支持单个连续 id 新目标，已经足够支撑演示。
- 扩展复杂编号会牵涉目标索引、矩阵兼容、可视化和测试。

建议写入局限性或后续展望。

### 4. 标准分布式 CBBA 或复杂拍卖共识机制

不建议做。

原因：

- 当前方法已经定位为 MCHA，即基于多维代价的启发式拍卖重分配方法。
- 若转向标准 CBBA，需要引入 winner table、bundle、共识迭代等机制。
- 会导致论文方法描述和代码实现大幅变化。
- 剩余时间内风险过高。

### 5. THREAT_ADDED 与 A* 实时绕行深度耦合

不建议做。

原因：

- 当前 `THREAT_ADDED` 已能基于任务链航段威胁积分触发局部重分配。
- 若引入 A* 判断是否可以绕行，会把事件分析变成路径规划问题。
- 会显著增加运行成本和调试复杂度。
- 路径规划不是本课题主要创新点。

建议保持当前设计：

- `THREAT_ADDED` 用于任务重分配触发。
- A* + B 样条用于最终航迹规划展示。
