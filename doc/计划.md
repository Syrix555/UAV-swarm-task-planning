# 毕设后续开发计划

当前时间：2026 年 4 月 26 日  
预计答辩时间：2026 年 5 月底

从当前时间线看，剩余可用于开发、调试、写论文、整理图表和准备答辩的时间大约为 4 周。因此后续目标不应再定义为“继续完善全部系统能力”，而应定义为：

> 稳定交付一条论文可讲、答辩可演示、结果可复现的无人集群协同任务规划主链路。

---

## 一、后续最高优先级工作

### 1. 补任务序列可视化连线

优先级：最高

当前算法已经支持任务序列：

```text
UAV -> T1 -> T2 -> T3
```

但部分可视化仍然使用旧矩阵画法：

```text
UAV -> T1
UAV -> T2
UAV -> T3
```

这会导致论文图和真实算法语义不一致，是目前最容易在答辩展示中暴露的问题。

后续应新增基于 `AssignmentPlan / UavTaskSequence` 的任务链绘图函数，至少保证：

- 预分配结果图能展示 UAV 的任务序列
- 重分配前后对比图能展示任务链变化
- 一架 UAV 承担多个任务时，图上表现为连续链路，而不是从 UAV 起点发散多条边

这项工作量相对可控，但展示收益很高，建议优先完成。

### 2. 稳定主演示流程

优先级：最高

应整理出一条稳定可复现的主流程：

```text
场景加载
-> PSO 预分配
-> 动态事件触发
-> MCHA 重分配
-> 最终航迹规划
-> 图表或动图展示
```

建议主展示事件控制在 3 类左右：

- `UAV_LOST`
- `TARGET_ADDED`
- `THREAT_ADDED`

这三类事件代表性较强，也更容易服务论文叙述。

`TARGET_DEMAND_INCREASED` 可以作为补充实验或备用展示，不建议继续围绕它做大量扩展。

### 3. 统一论文展示脚本使用任务序列版

优先级：高

当前旧矩阵版流程和任务序列版流程并存。为了避免不同脚本输出语义不一致，后续论文主展示脚本应优先使用：

- `AssignmentPlan`
- `analyze_plan_event_impact()`
- `run_mcha_for_plan()`

旧矩阵版可以继续保留作为兼容或历史对比，但不建议作为论文主展示结果。

### 4. 整理论文实验图表和指标

优先级：高

后续需要尽早固定实验图和统计指标，不要等代码全部写完再整理。

建议至少准备：

- PSO 收敛曲线
- 预分配任务序列图
- 重分配前后对比图
- 目标需求满足情况图
- 最终路径规划结果图
- 简单统计表：
  - 总航程或总代价
  - 任务满足率
  - 重分配变化数量
  - 受影响目标数量
  - 算法运行时间

这些内容比继续增加低优先级功能更直接服务论文和答辩。

## 二、可以做，但必须控制范围的工作

### 1. TARGET_DEMAND_INCREASED 展示

如果当前流程已经稳定，可以保留该事件作为补充实验。

但不建议继续围绕该事件做复杂扩展，因为它不是当前最需要解决的问题。只要能展示“目标需求增加后，MCHA 补充 UAV 执行任务”即可。

### 2. 路径规划接任务链

路径规划在本课题中属于完整链路展示的一部分，但不是主要创新点。

可以做的范围：

- 将最终任务链逐段交给 A*
- 对路径做 B 样条平滑
- 输出最终航迹图

不建议做的范围：

- 不要把路径规划改成复杂动态避障系统
- 不要让 `THREAT_ADDED` 和 A* 深度耦合
- 不要为了路径规划大幅修改任务分配模型

原因是本课题重点在预分配和重分配，路径规划只需要支撑“全链路可展示”即可。

### 3. 动图或视频演示

可以做，但应放在静态图稳定之后。

建议先保证：

- 每个关键阶段都能稳定出图
- 每张图含义清楚
- 主流程可重复运行

之后再考虑把图表或过程串成动图/视频。不要现在就投入太多时间做复杂动画系统，否则容易影响论文和核心实验整理。

## 三、不建议继续做的工作及原因

### 1. 完整 UavExecutionState 执行态重分配

不建议做。

原因：

- 剩余时间有限，执行态会显著扩大工程量
- 需要处理当前位置、当前时间、剩余弹药、剩余航程、已完成任务和未完成任务
- 会影响 `UAV_LOST`、`THREAT_ADDED`、MCHA 代价评估和可视化等多个模块
- 很容易引入新的边界问题，影响主流程稳定性

当前论文中应将已实现内容表述为：

> 任务序列方案层面的事件触发式局部重分配

而不是完整的执行过程中实时重规划。

### 2. 任务链中间插入、重排和交换优化

不建议做。

原因：

- 当前链尾追加策略已经足够支撑任务序列重分配
- 中间插入会显著增加代价评估复杂度
- 链内重排和跨 UAV 交换会影响 PSO、MCHA、测试和可视化
- 对本科毕设展示收益有限，但实现和调试风险较高

当前更适合保留“链尾追加”的简洁策略，并在论文中说明这是为了保证重分配快速性和可解释性。

### 3. 多目标同时新增、非连续 id 和删除后复用 id

不建议做。

原因：

- 这些属于工程边界条件，不是论文核心创新点
- 当前 `TARGET_ADDED` 支持单个连续 id 新目标，已经足够支撑演示
- 扩展复杂编号会牵涉目标索引、矩阵兼容、可视化和测试
- 对答辩展示收益很低

建议将这些内容写入局限性或后续展望。

### 4. 标准分布式 CBBA 或复杂拍卖共识机制

不建议做。

原因：

- 当前方法已经定位为 MCHA，即基于多维代价的启发式拍卖重分配方法
- 若转向标准分布式 CBBA，需要引入 winner table、bundle、共识迭代等机制
- 会导致论文方法描述和代码实现大幅变化
- 剩余时间内风险过高

当前应继续保持 MCHA 的问题导向定位，强调局部快速修复和多维代价可解释性。

### 5. THREAT_ADDED 与 A* 实时绕行深度耦合

不建议做。

原因：

- 当前 `THREAT_ADDED` 已经能基于任务链航段威胁积分触发局部重分配
- 若引入 A* 判断是否可以绕行，会把事件分析变成路径规划问题
- 会显著增加运行成本和调试复杂度
- 路径规划不是本课题主要创新点

建议保持当前设计：

- `THREAT_ADDED` 阶段用于任务重分配触发
- A* + B 样条用于最终航迹规划展示

## 四、时间安排建议

### 4 月 26 日至 5 月上旬

主要目标：修展示链路，固定主流程。

建议完成：

- 补任务序列可视化连线
- 统一主展示脚本走任务序列版
- 固定随机种子和代表性场景
- 确保 `UAV_LOST / TARGET_ADDED / THREAT_ADDED` 都能稳定出图
- 检查每个主展示事件是否满足目标需求

### 5 月上旬至 5 月中旬

主要目标：整理论文实验和文字材料。

建议完成：

- 整理实验图
- 整理指标表
- 写任务建模、约束建模和算法流程
- 写 PSO 预分配实验分析
- 写 MCHA 重分配实验分析
- 写路径规划展示分析
- 明确写出当前方法局限和后续展望

### 5 月中旬至答辩前

主要目标：稳定和展示，不再大改功能。

建议完成：

- 只修 bug，不增加大功能
- 准备答辩 PPT
- 制作必要的动图或视频
- 多次运行主流程，确保结果稳定
- 准备可能被问到的问题：
  - 为什么使用 PSO
  - 为什么 MCHA 不是标准 CBBA
  - 为什么任务序列只做链尾追加
  - 为什么路径规划只用 A* + B 样条
  - 为什么没有完整实时执行态重规划

---

## 五、当前阶段结论

现在仍然有时间把项目做成一个完整、可展示、论文能讲通的本科毕设项目。

但已经不适合继续做大架构升级。

后续最重要的不是“功能越多越好”，而是：

- 主流程稳定
- 图表可信
- 论文表述和代码实现一致
- 答辩展示能顺畅跑通

因此，下一步最应该优先完成的是：

1. 任务序列可视化
2. 主展示脚本稳定化
3. 实验图表和论文材料整理
