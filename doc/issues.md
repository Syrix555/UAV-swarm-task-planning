## 一、当前仍需关注的问题

### 1. 当前重分配不是完整执行过程中的实时重规划

当前任务序列版重分配更接近：

```text
预分配得到完整 AssignmentPlan
-> 事件发生
-> 在完整任务序列方案上做局部修复
```

但还没有真正接入 `UavExecutionState` 中的执行进度信息：

- `completed_task_count`
- `current_position`
- `current_time`
- `remaining_ammo`
- `remaining_range`

因此当前 `UAV_LOST` 任务序列版采用的是“整链释放”：

```text
UAV-0: [T2, T5, T9]
-> UAV-0 损失
-> 释放 [T2, T5, T9]
```

还没有实现更真实的“只释放未完成任务”：

```text
UAV-0: [T2, T5, T9]
completed_task_count = 1
-> 保留已完成 T2
-> 释放未完成 [T5, T9]
```

当前阶段不建议继续实现完整执行态重分配。论文中应避免写成已经实现“执行过程中实时重规划”，更准确的说法是：

> 任务序列方案层面的事件触发式局部重分配。

---

### 2. TARGET_ADDED 仍有工程边界

任务序列版 `TARGET_ADDED` 已经支持单个新增目标，并可处理 `required_uavs > 1` 的需求补位。

但当前仍存在以下边界：

- 暂不支持多目标同时新增。
- 暂不支持非连续目标 id。
- 暂不支持目标删除后再新增导致的 id 复用。
- 任务序列版仍要求新增目标 id 连续，即：

```text
target.id == len(battlefield.targets) - 1
```

这些属于工程边界条件，不是当前论文主线。建议不再继续扩展，只在论文局限性或后续展望中说明。

---

### 3. THREAT_ADDED 仍是方案层局部修复，不是完整威胁绕行重规划

任务序列版 `THREAT_ADDED` 已经完成第一版闭环。当前采用的是：

```text
新增威胁航段检测
-> 保留安全前缀
-> 释放受影响航段目标及其后续任务
-> MCHA 局部修复开放目标需求
```

该方案已经能够避免新增威胁后部分目标无人执行，并能输出完整的重分配结果和分析图。

仍需注意的局限：

- 未接入 `UavExecutionState`。
- 不能区分已完成任务、正在执行任务和未执行任务。
- 当前新增威胁检查基于完整任务链，而不是只检查未完成航段。
- 威胁影响判断使用直线航段威胁积分，没有结合 A* 绕行后的真实可达性。
- 一旦某航段超过威胁阈值，就释放该航段目标及后续任务，策略偏保守。
- `forbidden_pairs` 主要避免释放任务重新回到同一 UAV，不能保证新航段完全避开新增威胁。

论文中建议描述为：

> 新增威胁下的任务序列方案层局部重分配。

不要表述为完整实时路径重规划。

---

### 4. TARGET_DEMAND_DECREASED 暂不作为主线实现

`TARGET_DEMAND_DECREASED` 事件暂不作为任务序列版主流程的重点实现对象。

原因：

- 打击任务中更常见、更值得展示的是无人机损失、目标新增、目标需求增加和新增威胁。
- 目标需求降低通常意味着任务点数量、价值或饱和打击需求被重新确认，这往往依赖无人机近距离感知。
- 如果 UAV 已经接近任务点才发现“不需要这么多 UAV”，再将其抽离并重分配，收益可能不明显。
- 该事件会自然引入当前位置、已飞行距离、剩余航程、剩余弹药和当前时间等执行态问题。

论文中若提到该事件，建议说明本文主要关注更具代表性的增压型或风险型动态扰动，目标需求降低由于发生概率和重分配收益较低，暂不作为核心实验场景展开。

---

### 5. 旧矩阵版流程仍作为兼容存在

当前主展示流程应优先使用：

- `AssignmentPlan`
- `UavTaskSequence`
- `analyze_plan_event_impact()`
- `run_mcha_for_plan()`

旧矩阵版分配和重分配逻辑仍作为历史兼容存在，但不建议作为论文主展示流程。

尤其需要注意：

- 矩阵版 `TARGET_ADDED` 更适合单个新目标。
- 矩阵版 `TARGET_ADDED` 最好满足 `required_uavs = 1` 且 id 紧接旧目标数。
- 如果后续脚本混用矩阵版和任务序列版，可能出现图表语义不一致。

后续新增论文图或主演示脚本时，应优先走任务序列版。

### 6. 重分配代价变化 CSV 的解释需要谨慎

`cost_change.csv` 中包含：

```text
事件前预分配
事件后待修复
重分配后
```

其中“事件后待修复”可能出现总代价低于事件前的情况。这不一定说明方案更优，而可能是因为部分任务被释放后方案不完整。

解释该表时必须同时看：

- 未满足任务需求数量
- 目标需求满足率
- 任务槽位数量
- 活跃 UAV 数

论文中不要只用总代价判断“事件后待修复”是否更好。

---

### 7. MCHA 候选投标得分图需要配合文字解释

候选投标得分图展示的是每轮中每架 UAV 通过可行性和最低分过滤后的“当前最优投标”，不是所有 UAV-target 理论组合。

因此某些事件中会看到：

- 只有部分 UAV 参与投标。
- 某一轮中多个 UAV 同时中标。
- `THREAT_ADDED` 的某轮可能出现多个候选都被接受。

这是因为当前 MCHA 是开放目标需求修复过程，每个目标可以按剩余需求接收多个 UAV，并不是单目标单赢家拍卖。

论文图说明中应补充这一点，避免读者误解为“所有 UAV 都应该对所有目标竞价”。

### 8. 结果目录中可能保留旧版输出

当前脚本已经按较清晰的目录输出新图，例如：

如果需要最终归档，可以在答辩前单独整理一次 `results/`，但当前不建议为了清理结果文件引入额外风险。

## 三、后续最应该检查的事项

### 1. 路径规划是否已经接入任务序列

这是当前最值得继续推进的问题。

后续应确认：

- 是否能将重分配后的 `AssignmentPlan` 传给路径规划。
- 是否能按每架 UAV 的任务链逐段调用 A*。
- 是否能输出最终多 UAV 航迹图。
- 是否能处理某些航段因威胁区过密导致无路可走的问题。

### 2. 主演示脚本是否稳定

后续应整理一个固定场景、固定随机种子、固定事件的主演示流程，避免答辩时临时拼多个脚本。

### 3. 最终论文图表是否已经冻结

后续应明确哪些图进入论文，哪些只作为调试或备用，不要继续无限调图。

### 4. 论文表述是否与代码边界一致

特别要避免以下表述：

- 已实现完整实时重规划。
- 已实现标准分布式 CBBA。
- 已实现复杂动态避障。
- 支持任意目标新增/删除/id 重映射。

更准确的表述是：

- 任务序列方案层面的事件触发式局部重分配。
- 基于多维代价的启发式拍卖重分配方法。
- 基于直线航段威胁评估的新增威胁任务链修复。
- A* + 平滑用于最终航迹展示。
