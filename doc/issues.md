# ISSUES

## 1

  现在 available_uavs 在 decrease 场景下仍然会打印成全体 UAV：

  available_uavs: [0, 1, 2, ..., 23]

  这不影响正确性，但在“纯裁剪事件”里它的解释价值不大。
  不过这已经属于展示优化问题了，不是功能问题。

## 2

1. 矩阵版 TARGET_ADDED 只适合单个新目标，而且最好：

- required_uavs = 1
- id 紧接旧目标数

2. 还没做多目标同时新增
3. 任务序列版 TARGET_ADDED 已支持单个新目标的 required_uavs > 1，但矩阵版和可视化流程仍需继续验证
4. 还没做“删目标后再加目标”的复杂编号问题
5. 任务序列版 TARGET_ADDED 目前仍要求新增目标 id 连续，即 target.id == len(battlefield.targets) - 1
6. 任务序列版暂不支持非连续目标 id 映射、多目标同时新增、目标删除后复用编号等复杂场景

## 3

  预分配已经改成任务序列语义后，PSO 的代价评估按任务链累计计算：

  UAV 起点 -> T1 -> T2 -> T3

  也就是说，第二个任务的距离、威胁和到达时间应基于 T1 -> T2，而不是 UAV 起点 -> T2。

  但当前预分配可视化仍然使用旧的分配矩阵画法：

  UAV 起点 -> T1
  UAV 起点 -> T2
  UAV 起点 -> T3

  这会导致一架 UAV 执行多个任务时，图上从 UAV 起点发散出多条虚线，和真实任务序列不一致。

  后续应新增基于 AssignmentPlan / UavTaskSequence 的任务链绘图函数，例如：

  - UAV 起点连接到第一个任务
  - 第一个任务连接到第二个任务
  - 第二个任务连接到第三个任务

  旧的矩阵连线函数可以保留给重分配旧流程或对比图使用。

## 4

  当前任务序列版重分配还不是真正的“任务执行过程中”的实时重分配。

  目前实现更接近：

  预分配得到完整 AssignmentPlan
  -> 事件发生
  -> 在完整任务序列方案上做局部修复

  但还没有真正接入 UavExecutionState 中的执行进度信息：

  - completed_task_count
  - current_position
  - current_time
  - remaining_ammo
  - remaining_range

  因此当前 UAV_LOST 任务序列版采用的是“整链释放”：

  UAV-0: [T2, T5, T9]
  -> UAV-0 损失
  -> 释放 [T2, T5, T9]

  还没有实现更真实的“只释放未完成任务”：

  UAV-0: [T2, T5, T9]
  completed_task_count = 1
  -> 保留已完成 T2
  -> 释放未完成 [T5, T9]

  后续如果论文或演示要强调任务执行过程中的动态重分配，需要增加一个执行态扩展阶段：

  - 基于 UavExecutionState 更新每架 UAV 的当前位置、当前时间、剩余弹药和剩余航程
  - 事件分析只处理未完成任务链
  - MCHA 从 UAV 当前状态出发评估链尾追加代价
  - THREAT_ADDED 只检查未完成航段，而不是完整历史任务链

  在此之前，论文表述应避免写成已经完整实现“执行过程中实时重规划”，更准确的说法是“任务序列方案层面的事件触发式局部重分配”。

## 5

  TARGET_DEMAND_DECREASED 事件暂不作为第四阶段任务序列改造的重点实现对象。

  原因是该事件在打击任务中的优先级和现实收益都偏低：更常见、更值得展示的动态扰动是无人机损失、目标新增、目标需求增加和新增威胁。目标需求降低通常意味着任务点数量、价值或饱和打击需求已经被重新确认，而这种确认往往依赖无人机近距离感知。

  如果无人机已经接近任务点才发现“不需要这么多无人机”，此时再将其抽离并重分配，收益可能不明显，还会引入当前执行位置、已飞行距离、剩余航程、剩余弹药和当前时间等执行态问题。

  因此当前阶段先不实现任务序列版 TARGET_DEMAND_DECREASED。旧矩阵版已有基础支持，但任务序列版暂不接入主流程。

  若后续确实需要补充该事件，建议不要简单按完整任务链裁剪，而应结合 UavExecutionState 判断：

  - UAV 是否已经接近该目标
  - 该目标是否已经完成或正在执行
  - 删除该目标后是否值得继续重分配后续任务
  - 被释放 UAV 是否还有足够剩余航程、弹药和时间窗口参与其他任务

  论文中若提到该事件，应说明本文主要关注更具代表性的增压型或风险型动态扰动；目标需求降低事件由于发生概率和重分配收益较低，暂不作为核心实验场景展开。

## 6

  任务序列版 THREAT_ADDED 已经完成第一版闭环，但仍存在一些需要后续继续完善的边界问题。

  当前实现采用的是“新增威胁航段检测 + 后缀释放 + MCHA 局部修复”：

  UAV 起点 -> T1 -> T2 -> T3

  如果某条航段受到新增威胁显著影响，则保留该航段之前的安全前缀，释放该航段目标及其后的任务后缀，再由任务序列版 MCHA 对释放任务重新分配。

  该方案已经解决了旧矩阵版 THREAT_ADDED 中部分目标点无人执行的问题，并且通过 scenario_reallocation 场景校验可以保证任务序列版结果中目标需求被补齐。

  但它仍然不是完整真实执行态的威胁重规划，主要局限如下：

  - 仍然基于完整 AssignmentPlan 做方案层面的事件触发式修复，没有接入 UavExecutionState
  - 还不能区分已完成任务、正在执行任务和未执行任务
  - 当前新增威胁只检查完整任务链航段，而不是只检查未完成航段
  - 威胁影响判断使用直线航段威胁积分，没有结合 A* 路径绕行后的真实可达性
  - 一旦某航段超过威胁阈值，就释放该航段目标及后续任务，策略偏保守，可能释放过多任务
  - forbidden_pairs 只避免释放任务分回同一 UAV，不能保证重分配后的新航段一定完全避开新增威胁
  - 可视化输出已经切换为任务序列版重分配结果，但图上连线仍主要依赖旧矩阵画法，没有真正绘制 UAV -> T1 -> T2 的任务链路线

  后续如果继续优化 THREAT_ADDED，建议优先考虑两件事：

  - 先补任务序列可视化连线，让论文图和 AssignmentPlan 语义一致
  - 再考虑接入 UavExecutionState，把威胁事件从“完整计划局部修复”扩展为“执行过程中未完成任务链修复”

  在论文表述中，当前版本建议描述为“新增威胁下的任务序列方案层局部重分配”，不要表述为已经实现完整实时路径重规划。
