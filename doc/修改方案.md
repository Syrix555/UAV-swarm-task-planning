# 可以，我直接给你一版**围绕“支持 UAV 执行任务序列”**的完整实现方案。

我这里默认你当前采用的是：

- 不做 B+ 显式同步项
- 保留时间窗软约束
- 对协同目标通过较窄时间窗近似约束多机到达接近
- 当前阶段的核心改造目标只有一个：把系统从 ammo=1 的单任务分配框架，升级成 ammo>1 的任务序列分配框架
- 预分配算法仍然以**改进 PSO**为主体，链尾追加只作为任务序列解码、候选评估和局部修复的规则，不把预分配主算法改成纯贪心

---

## 一、先明确这次改造后的系统目标

改造前，你的系统本质上是：

- 每架 UAV 最多执行 1 个目标；
- 预分配输出是“UAV-Target 匹配关系”；
- 重分配处理的是“单个目标的重新指派”；
- 路径规划只需要处理“起点 → 单个目标”。

改造后，希望变成：

- 每架 UAV 最多执行 ammo 个目标；
- 预分配输出变成“UAV 的任务序列”；
- 重分配处理的是“任务序列的未执行部分”；
- 路径规划要处理“任务链逐段规划”；
- 时间窗约束要基于累计到达时刻来计算，而不是只算单跳。

也就是说，这次改造的核心不是“多加一个字段”，而是把系统的基本对象从：

▎ 单次打击任务

改成：

▎ 单机任务链 / 任务序列

---

## 二、建议的总体架构

我建议把整个流程明确成四层：

1. **目标需求层**

描述每个目标需要什么：
- 目标位置
- 目标价值
- required_uavs
- 时间窗
- 其他约束

2. **分配层**

决定：
- 哪些 UAV 参与哪个目标
- 每架 UAV 的任务序列是什么

输出的是：
- uav_id -> [task1, task2, ...]

3. **时序评估层**

根据任务序列计算：
- 每个 UAV 到各目标的预计到达时刻
- 是否满足时间窗
- 链式总代价

4. **路径规划层**

对任务序列逐段求路径：
- 起点 → 第一个目标
- 第一个目标 → 第二个目标
- …

这四层里，这次最主要动的是第 2、3 层，第 4 层主要是从“单段调用”变成“链式调用”。

## 三、最需要先改的数据结构

这是第一优先级，因为如果数据结构还停留在单任务语义，后面所有算法都不好改。

### 3.1 预分配结果：从单目标改成任务序列

你现在大概率是类似这种语义：

`assignment[uav_id] = target_id`

要改成：

`assignment[uav_id] = [target_id_1, target_id_2, ...]`

更推荐直接上 dataclass，而不是裸 list。

例如：

```python
@dataclass
class TaskNode:
    target_id: int
    order: int

@dataclass
class UavTaskSequence:
    uav_id: int
    tasks: list[TaskNode]
```

如果你希望后面少返工，可以一步到位一点：

```python
@dataclass
class TaskNode:
    target_id: int
    order: int
    planned_arrival_time: float = 0.0
    planned_service_time: float = 0.0
    estimated_path_length: float = 0.0
```

这样后面时间窗和路径代价都能自然挂进去。

### 3.2 目标侧索引：增加“谁参与了这个目标”

因为一个目标可能需要多架 UAV，所以你不能只从 UAV 侧看，还得能快速知道某目标当前被哪些 UAV 执行。

建议维护：

`target_assignees: dict[int, list[int]]`

比如：

```python
{
    3: [0, 2],
    5: [1],
    7: [0, 4, 6],
}
```

这个结构在以下场景很关键：

- 检查某目标是否已满足 required_uavs
- 目标需求变化时做重分配
- 可视化某目标的执行者集合
- 后续计算协同目标的到达时刻分布

### 3.3 重分配对象：区分已执行与未执行任务

原来单任务场景下，这个问题不明显；现在任务序列一来，这个状态必须清楚。

建议增加一个“执行进度”结构：

```python
@dataclass
class UavExecutionState:
    uav_id: int
    completed_task_count: int
    current_position: tuple[float, float]
    current_time: float
    remaining_ammo: int
    remaining_range: float
```

用途：
- 重分配时判断该 UAV 链上哪些任务已完成、哪些未完成
- 事件触发后从“当前状态”继续规划，而不是从初始状态重新来

### 3.4 建议新增一个统一的方案容器

建议你把分配结果统一收拢到一个结构里，不要散落成多个 dict。

```python
@dataclass
class AssignmentPlan:
    uav_task_sequences: dict[int, UavTaskSequence]
    target_assignees: dict[int, list[int]]
    total_cost: float
```

如果后续还要记录统计，可以继续加：
- 时间窗违约数
- 未满足目标数
- 总飞行距离
- 总收益等

## 四、预分配阶段要改什么

这是第一大块核心改造。

### 4.1 先把 ammo=1 的隐含假设全部找出来

你现在的预分配代码里，通常会有这些隐式单任务假设：

**假设 A**：一个 UAV 一旦被分配就退出竞争

例如类似：
- “已分配 UAV 不再参与后续目标分配”
- “每个 UAV 只保留一个最优目标”

这个要改。

**假设 B**：代价只算 uav -> target

现在要改成：
- 起点 / 当前链尾 -> 新目标

**假设 C**：航程检查只看单段

现在要改成：
- 任务链累计航程

**假设 D**：时间窗只看单目标到达

现在要改成：
- 由任务序列前缀累计得到的到达时刻

这几个隐含假设，基本就是这次改造的主战场。

### 4.2 ammo 的语义要重新定义

原来你这里的 ammo 很可能被等价理解成：

▎ 该 UAV 只能打 1 个目标

现在应改成：

▎ 该 UAV 最多可执行 ammo 次打击任务

因此约束从：

`uav 只能分配一个目标`

变成：

`len(uav_task_sequence.tasks) <= uav.ammo`

如果还考虑实际弹药消耗不是 1 发 / 目标，也可以以后扩展成：
- target.required_ammo
- uav.remaining_ammo -= required_ammo

但第一版建议先保持：
- 一个目标消耗一次打击能力
- 先把框架跑通

### 4.3 预分配输出改成“任务序列”

以前的预分配是在做 UAV-Target 匹配；
现在要让改进 PSO 的候选解能够被解码成 UAV 任务序列。

这里需要特别注意：

▎ **预分配主算法仍然是改进 PSO，不建议替换成纯贪心算法。**

更合适的做法是：
- PSO 仍然负责搜索候选分配方案
- 候选解解码后形成 `uav_id -> [task1, task2, ...]`
- 任务链的顺序构造采用简单的链尾追加规则
- 适应度评估调用任务链时序评估函数，而不是只算单跳距离

第一版建议采用最简单、最稳的任务序列构造策略：

**只允许“链尾追加”**

不要一上来做：
- 插入到任意位置
- 局部交换
- 序列优化

也就是说，第一版只做：

▎ 给某 UAV 当前任务链末尾追加一个新目标

也就是每次评估：

`append_cost(uav, current_sequence, target)`

而不是更复杂的：

`insert_cost(uav, current_sequence, target, pos)`

这样复杂度会小很多，且和本科毕设范围更匹配。
后续如果时间充裕，再考虑插入、交换或局部重排；当前阶段不建议一开始就做。

### 4.4 新的预分配评分逻辑

现在对 PSO 的某个候选解，不应只看静态 UAV-Target 距离，而应先把候选解解码成任务序列，再基于任务链累计代价计算适应度。

对于候选追加操作 `(uav, target)`，可以计算追加后的边际代价；对于完整粒子，则汇总所有 UAV 任务链的总代价。

建议边际代价至少包含 5 项：

1. **新增飞行代价**
   - 当前链尾到该目标的距离 / 预计飞行时间

2. **时间窗代价**
   - 若追加后预计到达时刻偏离目标时间窗，则惩罚

3. **航程可行性**
   - 若累计航程超出 range_left，直接判不可行或重罚

4. **弹药可行性**
   - 若任务数超 ammo，直接不可行

5. **目标价值 / 收益**
   - 高价值目标应能抵消部分代价

一个比较朴素的边际评分形式可以是：

`score(u, t) = reward(t) - \alpha \cdot \Delta distance - \beta \cdot time_window_penalty - \gamma \cdot risk`

如果你当前已有收益函数，那就不要推翻，只要把里面的单任务代价替换成任务链累计代价即可。

对于 PSO 适应度，可以继续保持“越小越好”的目标函数口径，例如：

`fitness = distance_cost + threat_cost + time_window_penalty + constraint_penalty - task_reward`

其中 `distance_cost`、`time_window_penalty` 和约束检查都应基于完整任务链，而不是只基于起点到目标的一跳。

### 4.5 对 required_uavs > 1 的处理

这里我建议保持简单：

对每个目标，仍按“需求人数”来填充

例如 required_uavs = 2：
- 就给它分配 2 架 UAV
- 这 2 架 UAV 可以各自都还有别的前后续任务

时间协调先只交给窄时间窗

不额外加同步项，只要求：
- 每个参与该目标的 UAV 在其时间窗附近到达
- 时间窗尽量设置窄一些

也就是说，预分配阶段只负责：
- 补足人数
- 控制任务链代价
- 尽量不违反时间窗

这就够了。

## 五、重分配阶段要改什么

这是第二大块核心改造，而且改动难度通常不低于预分配。

### 5.1 从“单目标重分配”改成“任务链重分配”

原来重分配通常是在处理：
- 某目标需要重新找执行 UAV
- 某 UAV 丢失后把它的目标交给别人

现在必须改成处理：
- 任务序列中的未完成部分

比如一架 UAV 原链为：

`[ T2, T5, T9 ]`

如果它在执行完 T2 后损失，那么要释放出来的不是整个链，而是：

`[ T5, T9 ]`

所以重分配的最小对象要变成：

▎ 未执行任务节点集合

### 5.2 重分配必须支持的三类操作

我建议第一版至少支持以下三类：

1. **截断**
   当 UAV 失效或退出任务时：
   - 保留已完成部分
   - 未完成部分释放回任务池

2. **删除**
   当目标取消、被摧毁、需求变化时：
   - 从若干 UAV 的任务链中删除该目标

3. **追加**
   将释放出来的新任务重新追加到其他 UAV 链尾

这三类操作足够支撑第一版。

### 5.3 第一版先不做复杂局部重排

比如：
- 把某 UAV 的第 3 个目标换到第 2 个
- 两架 UAV 交换链中的某些任务
- 对整条链做 TSP 风格顺序优化

这些都会让工作量明显膨胀。

所以我建议你当前阶段明确砍掉：

- 不做任意插入
- 不做跨 UAV 交换
- 不做复杂序列优化
- 重分配只做释放 + 重新链尾追加

这个取舍非常重要。

### 5.4 协同目标在重分配里的特殊点

如果某目标 required_uavs > 1，那在重分配时要特别注意：

**场景 1：参与者丢失**

例如某目标原本由 2 架 UAV 执行，其中 1 架失效：
- 那这个目标就少了一个参与者
- 需要补分配一架新的 UAV

**场景 2：需求变化**

例如 required_uavs 从 2 变成 3：
- 要再补 1 架 UAV

**场景 3：目标取消**

- 需要从多个 UAV 的任务序列中同时删掉这个目标

所以重分配阶段要从“谁丢了一个任务”升级成：

▎ 某个目标需求是否仍被满足
▎ 某些 UAV 的链是否需要局部修复

## 六、时间窗逻辑要改什么

这个部分你虽然不打算引入 B+，但其实还是要改，因为多任务链一来，到达时间的计算方式已经变了。

### 6.1 原来的时间窗可能是单跳时间

例如：
- UAV 从起点飞到目标
- 用这个时间检查时间窗

现在必须改成：

▎ 基于任务序列累计飞行时间，计算该 UAV 到达每个目标的时刻

即：
- arrival(T1) = start_time + travel(start, T1)
- arrival(T2) = arrival(T1) + service(T1) + travel(T1, T2)
- arrival(T3) = arrival(T2) + service(T2) + travel(T2, T3)

这样时间窗才有意义。

### 6.2 建议增加统一的任务链时序评估函数

非常建议做一个专门函数，例如：

`evaluate_uav_task_sequence(...)`

输入：
- UAV 初始状态
- 任务序列
- 路径/距离估计方式

输出：
- 每个任务节点的预计到达时间
- 总飞行距离
- 时间窗惩罚
- 是否超航程
- 是否超 ammo

这个函数会成为：
- 预分配评分的基础
- 重分配可行性检查的基础
- 路径规划后真实校正的基础

这是一个很值得单独抽出来的核心函数。

## 七、路径规划怎么接入

你现在路径规划模块已经比较完整了，所以这里不建议重写，只建议改调用粒度。

### 7.1 从“单段规划”改成“逐段规划”

原来可能是：
- 起点 → 目标

现在改成：
- 起点 → 第一个目标
- 第一个目标 → 第二个目标
- 第二个目标 → 第三个目标

最后把这些段拼起来。

### 7.2 建议分成两套代价

**预分配 / 重分配内部**

先用近似代价，避免太慢：
- 欧氏距离
- 或 LOS 距离估计
- 或缓存过的段代价

**最终结果生成**

对确定下来的任务链再调用真实路径规划器，得到：
- 真实航迹
- 真实路径长度
- 更准确的到达时刻

这个分层很重要，不然预分配一旦对所有候选都跑完整路径规划，开销会很大。

### 7.3 最好增加一个“航段缓存”

因为多任务链会频繁查询：
- A -> B
- B -> C
- A -> C

建议加一个缓存：

`segment_cache[(start_key, end_key)] = segment_result`

缓存内容可以包括：
- 距离
- 飞行时间
- 是否可行
- 路径点列

这样：
- 预分配更快
- 重分配也能复用

## 八、算法逻辑建议怎么改

这里我给你一版适合当前阶段的最小可行算法逻辑。

### 8.1 预分配：改进 PSO 生成任务序列

预分配阶段不建议改成纯贪心算法，因为论文主线已经确定为“改进 PSO 预分配算法”。

更合适的第一版做法是：

▎ **保留改进 PSO 作为搜索框架，把粒子解码和适应度评估改成任务序列语义。**

**Step 1：初始化**
- 所有 UAV 的任务链为空
- 所有目标的已分配人数为 0

**Step 2：调整 PSO 编码**

原来的排列型编码隐含了“同一 UAV 不能重复出现”，这不适合 ammo > 1。

新的编码需要允许：
- 同一 UAV 出现在多个任务槽位中
- 但同一 UAV 的出现次数不能超过 `ammo`
- 每个目标仍根据 `required_uavs` 占据对应数量的任务槽位

**Step 3：将粒子解码为任务序列**

解码时按照槽位顺序，把目标追加到对应 UAV 的任务链尾部。

这一步使用“链尾追加”规则，但它只是 PSO 候选解的解码方式，不是用贪心替代 PSO。

**Step 4：评估任务链适应度**

对每架 UAV 调用任务链评估函数，累计：
- 飞行距离
- 威胁代价
- 时间窗惩罚
- ammo / range 约束惩罚
- 目标收益

最终得到粒子的适应度值。

**Step 5：输出 AssignmentPlan**

PSO 结束后，不只输出旧的分配矩阵，还应输出或转换成 `AssignmentPlan`，方便后续重分配和可视化继续使用。

这种做法的优点是：
- 论文中仍然可以清楚写“改进 PSO”
- 代码上能接入任务序列
- 链尾追加保持实现简单
- 不需要引入插入、交换、局部重排等复杂操作

### 8.2 重分配：事件驱动的局部修复

建议第一版这样做：

**输入**
- 当前 AssignmentPlan
- 当前 UAV 执行状态
- 事件类型（UAV_LOST / TARGET_ADDED / TARGET_REMOVED / DEMAND_CHANGED）

**处理**
- 先识别受影响任务
- 从旧任务链里删除/截断
- 形成“待重新分配任务池”
- 对任务池中的任务执行“链尾追加式 MCHA 局部修复”

**输出**
- 更新后的 AssignmentPlan

这样能最大程度复用任务链评估逻辑，同时保留“预分配用改进 PSO、重分配用 MCHA”的论文结构。

## 九、建议的实现顺序

这个顺序很关键，我建议你不要跳。

### 第一阶段：先改数据结构

先做这些：

1. 单任务分配结果 → uav_task_sequences
2. 增加 target_assignees
3. 增加 UavExecutionState
4. 增加统一 AssignmentPlan

这一阶段结束后，即便算法还没完全改完，至少系统的“表达能力”已经从单任务升级到任务链了。

### 第二阶段：抽出任务链评估函数

实现统一的：
- 任务链累计距离
- 任务链累计时间
- 时间窗检查
- ammo 检查
- range 检查

这一步完成后，预分配和重分配就有共同依赖的底层能力。

### 第三阶段：改预分配

改成：
- 支持 ammo > 1
- 支持同一 UAV 在多个任务槽位中出现，但不超过 ammo
- 支持将 PSO 粒子解码为链尾追加式任务序列
- 支持 required_uavs > 1
- 输出任务序列 / AssignmentPlan，而不是只输出单目标映射

这是主功能成型阶段。

### 第四阶段：改重分配

支持：
- UAV 损失后的链截断
- 目标新增后的链尾追加
- 目标删除/需求变化后的局部修复

这一步之后系统基本能跑完整动态流程。

### 第五阶段：接路径规划

把最终任务链逐段丢给路径规划器，得到：
- 分段路径
- 整体航迹
- 更准确到达时间

### 第六阶段：补可视化与测试

建议重点补以下测试：

**预分配测试**

- required_uavs=2 时一个目标能被两架 UAV 分到
- 超 ammo 不会继续分配
- 超 range 会被拒绝

**重分配测试**
- UAV 损失后只释放未完成任务
- 目标新增后能追加到链尾
- 目标删除后会从相关 UAV 链中移除

**时间窗测试**
- 多任务链累计到达时刻正确
- 后续任务确实受前序任务影响

**路径规划测试**

- 总路径长度为各段累计

## 十三、任务序列改造阶段更新

当前任务序列相关改造已经完成了一个可运行的第一版闭环，整体目标是把原本“单架 UAV 只能执行一个目标”的模型，推进到“单架 UAV 可以在 ammo 和航程约束下按序执行多个目标”。

### 1. 已完成的主要工作

**数据结构层面**

- 已使用 `AssignmentPlan` 作为统一任务序列分配结果
- 已使用 `UavTaskSequence` 表达单架 UAV 的串行任务链
- 已使用 `TaskNode` 表达任务链中的目标节点
- 已通过 `target_assignees` 保留目标到 UAV 的反向索引
- 任务序列结果仍可投影回旧的 assignment matrix，兼容原有部分流程

**任务链评估层面**

- 已实现按任务顺序累计计算距离、到达时间和时间窗惩罚
- 已接入 ammo 约束和 range 约束
- 第二个及后续任务的代价已经按前序任务到当前任务计算，例如 T1 -> T2，而不是 UAV 起点 -> T2

**预分配层面**

- PSO 已支持 UAV 在 ammo 允许范围内承担多个任务槽位
- PSO 粒子可解码为链尾追加式任务序列
- 适应度评估已按任务链累计代价计算
- `run_pso(..., return_assignment_plan=True)` 已可返回 `AssignmentPlan`
- small / medium / hard 等场景已调整为 UAV 数量少于目标数量，更能体现一对多任务序列

**重分配层面**

- 已新增任务序列版 `PlanReallocationState`
- 已新增任务序列版 `run_mcha_for_plan()`
- MCHA 已支持在任务链尾部追加开放任务
- 边际得分已基于追加前后任务链代价变化计算
- 当前任务序列版已支持：
  - `TARGET_DEMAND_INCREASED`
  - `UAV_LOST`
  - `TARGET_ADDED`
  - `THREAT_ADDED`

**新增威胁事件层面**

- `THREAT_ADDED` 已支持按任务链航段检查新增威胁：
  - UAV 起点 -> T1
  - T1 -> T2
  - T2 -> T3
- 若某航段受新增威胁影响，则保留安全任务前缀，释放受影响任务后缀
- 已加入 `forbidden_pairs`，避免刚因威胁释放的任务又被分回原 UAV
- 重分配可视化中的新增威胁流程已切换为任务序列版结果，避免旧矩阵版出现部分目标无人执行的问题

### 2. 当前仍存在的不足

**可视化仍未完全任务序列化**

算法层面已经是：

UAV -> T1 -> T2 -> T3

但部分图仍然使用旧矩阵画法，视觉上可能表现为：

UAV -> T1
UAV -> T2
UAV -> T3

这会导致图示和真实任务链语义不一致。后续应优先补充基于 `AssignmentPlan / UavTaskSequence` 的任务链连线绘制。

**重分配还不是完整执行态重分配**

当前重分配更接近：

预分配得到完整 AssignmentPlan -> 事件发生 -> 在完整任务序列方案上做局部修复

它还没有真正接入 `UavExecutionState`，因此还不能区分：

- 已完成任务
- 正在执行任务
- 未执行任务

这意味着当前论文中应避免直接表述为“执行过程中实时重规划”，更准确的说法是“任务序列方案层面的事件触发式局部重分配”。

**MCHA 的任务插入策略仍然保守**

当前 MCHA 只做链尾追加：

- 不做中间插入
- 不做链内重排
- 不做跨 UAV 交换优化

这个取舍有利于解释和实现稳定，但重分配结果不一定是全局最优。

**THREAT_ADDED 仍是简化版**

当前新增威胁判断使用直线航段威胁积分，没有结合 A* 绕行后的真实可达性。

同时，一旦某航段超过威胁阈值，就释放该航段目标及其后的任务后缀，策略偏保守，可能释放过多任务。`forbidden_pairs` 只能避免任务分回原 UAV，不能保证新的任务链航段一定完全避开新增威胁。

**部分事件仍有边界限制**

- `TARGET_ADDED` 目前主要支持单个连续 id 新目标
- 尚未支持多目标同时新增
- 尚未支持非连续目标 id 映射
- 尚未支持目标删除后复用编号
- `TARGET_DEMAND_DECREASED` 已明确暂不作为任务序列版重点实现对象
- `TARGET_REMOVED` 和 `TARGET_VALUE_CHANGED` 不作为当前核心实验事件

### 3. 接下来行动评估

我认为接下来不应该优先继续堆复杂算法，而应优先保证“论文展示链路”和“任务序列语义”一致。

**第一优先级：补任务序列可视化连线**

预分配图和重分配图都应能按任务链绘制：

UAV -> T1 -> T2 -> T3

这是当前最容易在论文图和答辩展示中暴露问题的地方，工作量相对可控，收益很高。

**第二优先级：梳理稳定演示主流程**

建议形成一条明确的展示链路：

场景加载
-> PSO 预分配
-> 事件触发
-> MCHA 重分配
-> 最终航迹规划
-> 图表或动图展示

这比继续扩展低优先级事件更适合本科毕设展示。

**第三优先级：统一论文展示脚本使用任务序列版**

旧矩阵版流程可以继续保留作为兼容或历史对比，但论文主展示脚本应优先使用 `AssignmentPlan` 和任务序列版 MCHA，避免不同脚本输出语义不一致。

**第四优先级：暂缓执行态重分配**

除非论文必须强调“执行过程中实时重规划”，否则暂时不建议接入完整 `UavExecutionState`。

执行态会引入当前位置、当前时间、剩余弹药、剩余航程、已完成任务等新变量，复杂度明显上升，容易影响主线完成度。

### 4. 当前阶段结论

任务序列核心改造目前已经达到：

- 算法能跑
- 事件能接
- 测试能保
- 展示基本能用

下一阶段最应该做的是补齐可视化和演示链路，让代码实现、论文描述和最终图示保持一致。
