# 可以，我直接给你一版**围绕“支持 UAV 执行任务序列”**的完整实现方案。

我这里默认你当前采用的是：

- 不做 B+ 显式同步项
- 保留时间窗软约束
- 对协同目标通过较窄时间窗近似约束多机到达接近
- 当前阶段的核心改造目标只有一个：把系统从 ammo=1 的单任务分配框架，升级成 ammo>1 的任务序列分配框架

---

## 一、先明确这次改造后的系统目标

改造前，你的系统本质上是：

- 每架 UAV 最多执行 1 个目标；
- 预分配输出是“UAV-Target 匹配关系”；
- 重分配处理的是“单个目标的重新指派”；
- 路径规划只需要处理“起点 → 单个目标”。

改造后，希望变成：

- 每架 UAV 最多执行 ammo 个目标；
- 预分配输出变成“UAV 的任务序列”；
- 重分配处理的是“任务序列的未执行部分”；
- 路径规划要处理“任务链逐段规划”；
- 时间窗约束要基于累计到达时刻来计算，而不是只算单跳。

也就是说，这次改造的核心不是“多加一个字段”，而是把系统的基本对象从：

▎ 单次打击任务

改成：

▎ 单机任务链 / 任务序列

---

## 二、建议的总体架构

我建议把整个流程明确成四层：

1. **目标需求层**

描述每个目标需要什么：
- 目标位置
- 目标价值
- required_uavs
- 时间窗
- 其他约束

2. **分配层**

决定：
- 哪些 UAV 参与哪个目标
- 每架 UAV 的任务序列是什么

输出的是：
- uav_id -> [task1, task2, ...]

3. **时序评估层**

根据任务序列计算：
- 每个 UAV 到各目标的预计到达时刻
- 是否满足时间窗
- 链式总代价

4. **路径规划层**

对任务序列逐段求路径：
- 起点 → 第一个目标
- 第一个目标 → 第二个目标
- …

这四层里，这次最主要动的是第 2、3 层，第 4 层主要是从“单段调用”变成“链式调用”。

## 三、最需要先改的数据结构

这是第一优先级，因为如果数据结构还停留在单任务语义，后面所有算法都不好改。

### 3.1 预分配结果：从单目标改成任务序列

你现在大概率是类似这种语义：

`assignment[uav_id] = target_id`

要改成：

`assignment[uav_id] = [target_id_1, target_id_2, ...]`

更推荐直接上 dataclass，而不是裸 list。

例如：

```python
@dataclass
class TaskNode:
    target_id: int
    order: int

@dataclass
class UavTaskSequence:
    uav_id: int
    tasks: list[TaskNode]
```

如果你希望后面少返工，可以一步到位一点：

```python
@dataclass
class TaskNode:
    target_id: int
    order: int
    planned_arrival_time: float = 0.0
    planned_service_time: float = 0.0
    estimated_path_length: float = 0.0
```

这样后面时间窗和路径代价都能自然挂进去。

### 3.2 目标侧索引：增加“谁参与了这个目标”

因为一个目标可能需要多架 UAV，所以你不能只从 UAV 侧看，还得能快速知道某目标当前被哪些 UAV 执行。

建议维护：

`target_assignees: dict[int, list[int]]`

比如：

```python
{
    3: [0, 2],
    5: [1],
    7: [0, 4, 6],
}
```

这个结构在以下场景很关键：

- 检查某目标是否已满足 required_uavs
- 目标需求变化时做重分配
- 可视化某目标的执行者集合
- 后续计算协同目标的到达时刻分布

### 3.3 重分配对象：区分已执行与未执行任务

原来单任务场景下，这个问题不明显；现在任务序列一来，这个状态必须清楚。

建议增加一个“执行进度”结构：

```python
@dataclass
class UavExecutionState:
    uav_id: int
    completed_task_count: int
    current_position: tuple[float, float]
    current_time: float
    remaining_ammo: int
    remaining_range: float
```

用途：
- 重分配时判断该 UAV 链上哪些任务已完成、哪些未完成
- 事件触发后从“当前状态”继续规划，而不是从初始状态重新来

### 3.4 建议新增一个统一的方案容器

建议你把分配结果统一收拢到一个结构里，不要散落成多个 dict。

```python
@dataclass
class AssignmentPlan:
    uav_task_sequences: dict[int, UavTaskSequence]
    target_assignees: dict[int, list[int]]
    total_cost: float
```

如果后续还要记录统计，可以继续加：
- 时间窗违约数
- 未满足目标数
- 总飞行距离
- 总收益等

## 四、预分配阶段要改什么

这是第一大块核心改造。

### 4.1 先把 ammo=1 的隐含假设全部找出来

你现在的预分配代码里，通常会有这些隐式单任务假设：

**假设 A**：一个 UAV 一旦被分配就退出竞争

例如类似：
- “已分配 UAV 不再参与后续目标分配”
- “每个 UAV 只保留一个最优目标”

这个要改。

**假设 B**：代价只算 uav -> target

现在要改成：
- 起点 / 当前链尾 -> 新目标

**假设 C**：航程检查只看单段

现在要改成：
- 任务链累计航程

**假设 D**：时间窗只看单目标到达

现在要改成：
- 由任务序列前缀累计得到的到达时刻

这几个隐含假设，基本就是这次改造的主战场。

### 4.2 ammo 的语义要重新定义

原来你这里的 ammo 很可能被等价理解成：

▎ 该 UAV 只能打 1 个目标

现在应改成：

▎ 该 UAV 最多可执行 ammo 次打击任务

因此约束从：

`uav 只能分配一个目标`

变成：

`len(uav_task_sequence.tasks) <= uav.ammo`

如果还考虑实际弹药消耗不是 1 发 / 目标，也可以以后扩展成：
- target.required_ammo
- uav.remaining_ammo -= required_ammo

但第一版建议先保持：
- 一个目标消耗一次打击能力
- 先把框架跑通

### 4.3 预分配输出改成“构造任务序列”

以前的预分配是在做匹配；
现在更接近在做增量构造任务链。

我建议第一版采用最简单、最稳的策略：

**只允许“链尾追加”**

不要一上来做：
- 插入到任意位置
- 局部交换
- 序列优化

第一版只做：

▎ 给某 UAV 当前任务链末尾追加一个新目标

也就是每次评估：

`append_cost(uav, current_sequence, target)`

而不是更复杂的：

`insert_cost(uav, current_sequence, target, pos)`

这样复杂度会小很多，且和本科毕设范围更匹配。

### 4.4 新的预分配评分逻辑

现在对某个候选分配 (uav, target)，不应只看静态距离，而应看追加后的边际代价。

建议边际代价至少包含 5 项：

1. **新增飞行代价**
   - 当前链尾到该目标的距离 / 预计飞行时间

2. **时间窗代价**
   - 若追加后预计到达时刻偏离目标时间窗，则惩罚

3. **航程可行性**
   - 若累计航程超出 range_left，直接判不可行或重罚

4. **弹药可行性**
   - 若任务数超 ammo，直接不可行

5. **目标价值 / 收益**
   - 高价值目标应能抵消部分代价

一个比较朴素的形式可以是：

`score(u, t) = reward(t) - \alpha \cdot \Delta distance - \beta \cdot time_window_penalty - \gamma \cdot risk`

如果你当前已有收益函数，那就不要推翻，只要把里面的单任务代价替换成任务链增量代价即可。

### 4.5 对 required_uavs > 1 的处理

这里我建议保持简单：

对每个目标，仍按“需求人数”来填充

例如 required_uavs = 2：
- 就给它分配 2 架 UAV
- 这 2 架 UAV 可以各自都还有别的前后续任务

时间协调先只交给窄时间窗

不额外加同步项，只要求：
- 每个参与该目标的 UAV 在其时间窗附近到达
- 时间窗尽量设置窄一些

也就是说，预分配阶段只负责：
- 补足人数
- 控制任务链代价
- 尽量不违反时间窗

这就够了。

## 五、重分配阶段要改什么

这是第二大块核心改造，而且改动难度通常不低于预分配。

### 5.1 从“单目标重分配”改成“任务链重分配”

原来重分配通常是在处理：
- 某目标需要重新找执行 UAV
- 某 UAV 丢失后把它的目标交给别人

现在必须改成处理：
- 任务序列中的未完成部分

比如一架 UAV 原链为：

`[ T2, T5, T9 ]`

如果它在执行完 T2 后损失，那么要释放出来的不是整个链，而是：

`[ T5, T9 ]`

所以重分配的最小对象要变成：

▎ 未执行任务节点集合

### 5.2 重分配必须支持的三类操作

我建议第一版至少支持以下三类：

1. **截断**
   当 UAV 失效或退出任务时：
   - 保留已完成部分
   - 未完成部分释放回任务池

2. **删除**
   当目标取消、被摧毁、需求变化时：
   - 从若干 UAV 的任务链中删除该目标

3. **追加**
   将释放出来的新任务重新追加到其他 UAV 链尾

这三类操作足够支撑第一版。

### 5.3 第一版先不做复杂局部重排

比如：
- 把某 UAV 的第 3 个目标换到第 2 个
- 两架 UAV 交换链中的某些任务
- 对整条链做 TSP 风格顺序优化

这些都会让工作量明显膨胀。

所以我建议你当前阶段明确砍掉：

- 不做任意插入
- 不做跨 UAV 交换
- 不做复杂序列优化
- 重分配只做释放 + 重新链尾追加

这个取舍非常重要。

### 5.4 协同目标在重分配里的特殊点

如果某目标 required_uavs > 1，那在重分配时要特别注意：

**场景 1：参与者丢失**

例如某目标原本由 2 架 UAV 执行，其中 1 架失效：
- 那这个目标就少了一个参与者
- 需要补分配一架新的 UAV

**场景 2：需求变化**

例如 required_uavs 从 2 变成 3：
- 要再补 1 架 UAV

**场景 3：目标取消**

- 需要从多个 UAV 的任务序列中同时删掉这个目标

所以重分配阶段要从“谁丢了一个任务”升级成：

▎ 某个目标需求是否仍被满足
▎ 某些 UAV 的链是否需要局部修复

## 六、时间窗逻辑要改什么

这个部分你虽然不打算引入 B+，但其实还是要改，因为多任务链一来，到达时间的计算方式已经变了。

### 6.1 原来的时间窗可能是单跳时间

例如：
- UAV 从起点飞到目标
- 用这个时间检查时间窗

现在必须改成：

▎ 基于任务序列累计飞行时间，计算该 UAV 到达每个目标的时刻

即：
- arrival(T1) = start_time + travel(start, T1)
- arrival(T2) = arrival(T1) + service(T1) + travel(T1, T2)
- arrival(T3) = arrival(T2) + service(T2) + travel(T2, T3)

这样时间窗才有意义。

### 6.2 建议增加统一的任务链时序评估函数

非常建议做一个专门函数，例如：

`evaluate_uav_task_sequence(...)`

输入：
- UAV 初始状态
- 任务序列
- 路径/距离估计方式

输出：
- 每个任务节点的预计到达时间
- 总飞行距离
- 时间窗惩罚
- 是否超航程
- 是否超 ammo

这个函数会成为：
- 预分配评分的基础
- 重分配可行性检查的基础
- 路径规划后真实校正的基础

这是一个很值得单独抽出来的核心函数。

## 七、路径规划怎么接入

你现在路径规划模块已经比较完整了，所以这里不建议重写，只建议改调用粒度。

### 7.1 从“单段规划”改成“逐段规划”

原来可能是：
- 起点 → 目标

现在改成：
- 起点 → 第一个目标
- 第一个目标 → 第二个目标
- 第二个目标 → 第三个目标

最后把这些段拼起来。

### 7.2 建议分成两套代价

**预分配 / 重分配内部**

先用近似代价，避免太慢：
- 欧氏距离
- 或 LOS 距离估计
- 或缓存过的段代价

**最终结果生成**

对确定下来的任务链再调用真实路径规划器，得到：
- 真实航迹
- 真实路径长度
- 更准确的到达时刻

这个分层很重要，不然预分配一旦对所有候选都跑完整路径规划，开销会很大。

### 7.3 最好增加一个“航段缓存”

因为多任务链会频繁查询：
- A -> B
- B -> C
- A -> C

建议加一个缓存：

`segment_cache[(start_key, end_key)] = segment_result`

缓存内容可以包括：
- 距离
- 飞行时间
- 是否可行
- 路径点列

这样：
- 预分配更快
- 重分配也能复用

## 八、算法逻辑建议怎么改

这里我给你一版适合当前阶段的最小可行算法逻辑。

### 8.1 预分配：贪心构造任务链

建议第一版这样做：

**Step 1：初始化**
- 所有 UAV 的任务链为空
- 所有目标的已分配人数为 0

**Step 2：按某种优先级遍历目标**

优先级可按：
- 目标价值高优先
- 时间窗更紧优先
- required_uavs 更高优先
- 或几项加权组合

**Step 3：为当前目标挑选所需 UAV**

对每个还需要补的人数：
- 枚举可用 UAV
- 计算把该目标追加到其链尾的边际代价
- 选最优 UAV
- 更新其任务链和状态

**Step 4：直到所有目标处理完**

若某目标始终补不满：
- 记为未完全满足
- 在结果中给出惩罚或失败标记

这是本科毕设里很常见也很稳的路线。

### 8.2 重分配：事件驱动的局部修复

建议第一版这样做：

**输入**
- 当前 AssignmentPlan
- 当前 UAV 执行状态
- 事件类型（UAV_LOST / TARGET_ADDED / TARGET_REMOVED / DEMAND_CHANGED）

**处理**
- 先识别受影响任务
- 从旧任务链里删除/截断
- 形成“待重新分配任务池”
- 对任务池中的任务重新执行“链尾追加式贪心分配”

**输出**
- 更新后的 AssignmentPlan

这样能最大程度复用预分配逻辑，避免写两套完全不同的算法。

## 九、建议的实现顺序

这个顺序很关键，我建议你不要跳。

### 第一阶段：先改数据结构

先做这些：

1. 单任务分配结果 → uav_task_sequences
2. 增加 target_assignees
3. 增加 UavExecutionState
4. 增加统一 AssignmentPlan

这一阶段结束后，即便算法还没完全改完，至少系统的“表达能力”已经从单任务升级到任务链了。

### 第二阶段：抽出任务链评估函数

实现统一的：
- 任务链累计距离
- 任务链累计时间
- 时间窗检查
- ammo 检查
- range 检查

这一步完成后，预分配和重分配就有共同依赖的底层能力。

### 第三阶段：改预分配

改成：
- 支持 ammo > 1
- 支持链尾追加
- 支持 required_uavs > 1
- 输出任务序列而不是单目标映射

这是主功能成型阶段。

### 第四阶段：改重分配

支持：
- UAV 损失后的链截断
- 目标新增后的链尾追加
- 目标删除/需求变化后的局部修复

这一步之后系统基本能跑完整动态流程。

### 第五阶段：接路径规划

把最终任务链逐段丢给路径规划器，得到：
- 分段路径
- 整体航迹
- 更准确到达时间

### 第六阶段：补可视化与测试

这个阶段必须做，不然你后面很难验证。

建议重点补以下测试：

**预分配测试**
- ammo=2 时一架 UAV 能得到两个目标
- required_uavs=2 时一个目标能被两架 UAV 分到
- 超 ammo 不会继续分配
- 超 range 会被拒绝

**重分配测试**
- UAV 损失后只释放未完成任务
- 目标新增后能追加到链尾
- 目标删除后会从相关 UAV 链中移除

**时间窗测试**
- 多任务链累计到达时刻正确
- 后续任务确实受前序任务影响

**路径规划测试**
- 任务链能逐段生成路径
- 总路径长度为各段累计

## 十、工作量评估

如果按我上面这个“最小可行版”做，工作量我判断是：

**中等偏上，主要难在“改旧假设”**

真正费时间的不是新增几个函数，而是：
- 找出所有默认 ammo=1 的地方
- 把单任务语义替换成任务链语义
- 不让旧逻辑悄悄出错

复杂度主要来自三点：

1. 数据结构整体升级
2. 代价评估从单跳变成链式累计
3. 重分配从“单目标修补”变成“任务链局部修补”

但仍然是可控的。

因为你已经明确砍掉了这些高复杂度项：
- 不做显式同步项
- 不做复杂时序协同
- 不做任意位置插入
- 不做链内复杂重排
- 不做跨 UAV 交换优化

这个砍法非常关键，它让问题仍然维持在毕设可实现范围内。

## 十一、我建议你这一版的最终定位

你现在这一版最适合定义成：

▎ 在保留目标时间窗约束与动态重分配机制的基础上，将原有单任务分配模型扩展为支持弹药约束下多任务串行执行的任务序列分配模型。

如果再具体一点，可以说成：

▎ 每架无人机可在弹药和航程约束下执行多个目标，预分配阶段生成任务序列，重分配阶段对未完成任务链进行局部修复，协同目标通过较窄时间窗近似保证多机到达时刻的一致性。

这个表述我觉得和你当前的取舍是匹配的。

## 十二、最后给你一个最简洁的结论

如果只抓重点，这次要改的核心就是四件事：

1. **数据结构改成任务序列**
   - uav -> [task1, task2, ...]

2. **预分配改成链尾追加式构造**
   - 不再是一次性单目标匹配

3. **重分配改成未完成任务链的局部修复**
   - 不再只是单个目标转移

4. **时间与路径代价改成链式累计**
   - 不再只算起点到目标的一跳
